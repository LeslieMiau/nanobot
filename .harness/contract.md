# 冲刺合约 — 第 1 轮

## 构建范围

本轮将实现并验证以下功能：

- 功能 #1: 可观测的轮询进度
- 功能 #2: 卡死判定
- 功能 #3: 自动恢复
- 功能 #4: 误重启控制
- 功能 #5: 运行态可验证性
- 功能 #6: 回归测试
- 功能 #7: 真实 smoke 验证

## 成功标准

### 功能 #1: 可观测的轮询进度

- [ ] Telegram 运行态能记录最近一次轮询开始和完成的时间。
- [ ] 长轮询即使没有收到新消息，只要请求持续正常完成，也会被视为“有推进”。
- [ ] `/status` 或 `channels status` 输出中能看到轮询推进相关的状态信息。

### 功能 #2: 卡死判定

- [ ] 当轮询任务仍存活，但单次 `getUpdates` 长时间不返回时，系统会将其判定为卡死。
- [ ] 正常空闲但轮询仍周期性完成时，不会被判定为卡死。
- [ ] 卡死判定能给出明确原因，供日志和状态输出使用。

### 功能 #3: 自动恢复

- [ ] 轮询卡死后，watchdog 会触发 Telegram channel 自恢复，而不是继续将该状态视为健康。
- [ ] 自恢复成功后，运行态状态会清晰反映恢复后的新状态。
- [ ] 自恢复失败时，会沿用现有升级路径，不静默吞掉失败。

### 功能 #4: 误重启控制

- [ ] 单次短暂网络波动不会直接触发卡死恢复。
- [ ] 正常空闲窗口不会触发重复恢复。
- [ ] 同一时间最多只有一个恢复动作生效，避免恢复风暴。

### 功能 #5: 运行态可验证性

- [ ] 运行态输出能解释“为什么当前被视为健康 / 可疑 / 已恢复”。
- [ ] 修复后可从本机 gateway 日志或状态输出中确认 Telegram 轮询是否真实推进。

### 功能 #6: 回归测试

- [ ] 新增或更新测试，覆盖正常轮询推进场景。
- [ ] 新增或更新测试，覆盖轮询任务存活但请求卡死的场景。
- [ ] 新增或更新测试，覆盖卡死恢复不会误判正常空闲的场景。

### 功能 #7: 真实 smoke 验证

- [ ] 运行针对性 pytest 子集，证明回归保护有效。
- [ ] 在真实 gateway 进程上完成至少一轮运行态验证，确认 Telegram 通道状态输出和恢复逻辑没有明显回归。

## 验证方式

- 单元与集成层：运行 `tests/channels/test_telegram_channel.py` 与状态展示相关测试。
- 运行态层：按现有 tmux gateway 流程重启或观察 Telegram 通道，检查状态输出与日志。
- 证据层：记录本轮修改后的测试结果与关键运行态日志片段。
