# Telegram 长轮询稳定性修复规格书

## 愿景

让 nanobot 的 Telegram 通道在长时间运行、代理链路抖动、更新流短暂停顿或局部假死时，仍能可靠接收消息并自动恢复。

本次修复不是新增产品功能，而是针对现有 Telegram 通道的稳定性补强。核心目标是把“进程还活着但长轮询不再前进”的故障从不可见状态变成可观测、可判定、可恢复、可验证的状态。

## 功能列表

### 1. 可观测的轮询进度

系统需要能表达长轮询是否“真的在推进”，而不只是“相关任务还存活”。

验收标准：
- 能在运行态区分“正常空闲”和“长时间无进展”。
- 能展示最近一次有效推进的时间、状态和摘要。
- 能在日志或状态输出中看到轮询是否持续推进的判断依据。

### 2. 卡死判定

系统需要识别“任务活着但更新流停滞”的假死状态，并把它和普通网络波动、低消息量场景区分开。

验收标准：
- 长时间没有新增更新推进时，能触发卡死判定。
- 只有任务存活、没有进展，不应被误判为健康。
- 低流量但仍持续推进的场景不应被误判为故障。

### 3. 自动恢复

一旦确认轮询卡死，系统应能自动恢复到可接收消息的状态，尽量减少人工介入。

验收标准：
- 卡死后能触发恢复动作，并使通道重新进入可接收消息状态。
- 恢复过程有明确状态变化，不会静默失败。
- 恢复后应能继续接收后续入站消息。

### 4. 误重启控制

恢复机制不能因为短暂波动或正常空闲而频繁重启。

验收标准：
- 不因正常无消息窗口触发重复恢复。
- 不因单次短暂探测失败立即进入反复重启循环。
- 对同一故障应有节流或去抖策略，避免恢复风暴。

### 5. 运行态可验证性

修复后的行为必须能在真实运行环境里被证明，而不是只在静态代码层面看起来合理。

验收标准：
- 运行时能通过日志、状态或监控信息确认当前轮询健康状况。
- 能在本机环境里观察到故障识别与恢复过程。
- 能从运行态证据判断“消息为什么没有进入当前进程”。

### 6. 回归测试

必须为“进度推进、卡死识别、恢复动作、误判控制”建立回归保护。

验收标准：
- 有覆盖正常推进与卡死停滞的测试。
- 有覆盖自动恢复成功与恢复失败的测试。
- 有覆盖空闲但健康场景的测试，防止误报。

### 7. 真实 smoke 验证

需要用真实进程和真实 Telegram 链路完成一次端到端验证，证明修复不只是测试通过。

验收标准：
- 可以在本机真实启动服务并观察 Telegram 通道工作状态。
- 可以在长时间运行或模拟停滞后验证恢复结果。
- 可以用实际入站/出站消息证明修复有效。

## 技术栈

- 语言与运行时：Python
- CLI 入口：`nanobot`
- Telegram 通道库：`python-telegram-bot[socks] >=22.6,<23.0`
- 现有测试体系：`tests/channels`、`tests/cli`、`tests/agent`
- 本地初始化与校验：`init.sh`
- 运行环境特征：固定本机代理链路，现有 gateway 进程模型保持不变

## 架构

### 1. 观测层

负责记录轮询是否持续推进，以及推进的最新时间、最新状态和最近一次有效活动的摘要。这个层只回答“现在是否还在前进”，不直接决定是否恢复。

### 2. 判定层

负责综合任务存活、最近推进时间、更新流停滞时长和恢复历史，判断当前是健康、可疑还是已卡死。判定逻辑必须能解释，便于排障与审计。

### 3. 恢复层

负责在确认卡死后执行恢复动作，并把结果反馈回运行态状态。恢复层应避免多路并发重复触发，确保同一时刻只有一个权威恢复动作在进行。

### 4. 运行态输出层

负责把健康、可疑、卡死、恢复中、恢复成功、恢复失败等状态对外呈现出来，让 Telegram、CLI 和日志看到一致的结论。

### 5. 验证层

负责把上述能力落到测试和真实 smoke 中，确保可以重复验证故障判定准确性与恢复结果。

## 评估标准

### 1. 故障检测准确性

- 能识别真正的长轮询停滞。
- 不把正常空闲误判成故障。
- 不把短暂抖动误判成卡死。

### 2. 恢复可靠性

- 确认卡死后，系统能恢复接收消息。
- 恢复动作可重复验证，失败时有明确失败状态。
- 恢复后不会立刻回到同一假死状态而无感知。

### 3. 误重启控制

- 不能在健康运行时频繁重启。
- 不能因为观测信号不足而扩大故障影响。
- 不能产生恢复循环。

### 4. 日志与状态可观测性

- 运行态能看出轮询是否在推进。
- 运行态能看出为什么触发了恢复。
- 运行态能看出恢复是否成功以及后续是否恢复入站。

### 5. 当前环境可验证性

- 本机即可完成关键验证，不依赖抽象假设。
- 能通过真实进程、真实链路和真实消息确认结果。
- 测试与 smoke 的结论应一致，不允许只在单一层面“看起来成功”。

## 约束

- 只针对 Telegram 通道稳定性做定向修复，不扩展为跨通道架构重构。
- 不改变“单一权威轮询通道”的基本运行模型，也不引入额外的独立长轮询实体。
- 不把“任务存活”或单次探针成功当作健康完成条件。
- 不以删除断言、跳过测试或降低验证强度来换取通过。
- 不依赖外部不可控环境作为唯一验证手段，必须保留本机可重复验证路径。
- 不扩展到全局代理治理、DNS 重构或与本问题无关的服务编排变更，除非它们是修复所必需且已被证明不可替代。
